How Certificate Transparency Discovery Works

Certificate Transparency (CT) is an open framework for monitoring and auditing SSL/TLS certificates. Every certificate issued by a trusted CA must be logged to at least two public CT logs before browsers accept it.

1

CT Log Query

We query crt.sh for all certificates matching %.domain.com. This returns every certificate that includes a SAN for any subdomain of the target.

2

Subdomain Extraction

From each certificate, we extract the Common Name (CN) and all Subject Alternative Names (SANs). A single certificate can list dozens of subdomains.

3

Deduplication & Metadata

Subdomains are deduplicated. For each, we track earliest cert start date (first seen), latest cert expiry (last seen), and certificate count. Wildcards are excluded.

4

Limitations

CT-based discovery only finds subdomains with certificates. HTTP-only services and very new subdomains (not yet logged) won't appear. Results capped at 500.

Data Source & Standards

All data comes from public CT log infrastructure. No traffic is sent to the target domain.

What is Certificate Transparency subdomain discovery?

Certificate Transparency (CT) logs are public records of every SSL/TLS certificate issued by trusted Certificate Authorities. When a certificate is issued for a subdomain, it gets logged. By querying these logs, we can discover all subdomains that have ever had a certificate issued — revealing staging servers, internal tools, API endpoints, and forgotten services.

Is this subdomain finder free?

Yes, completely free with no signup required. The tool queries public Certificate Transparency logs via crt.sh, which is an open-source CT log search engine.

Will this find all subdomains?

This tool finds subdomains that have had SSL/TLS certificates issued for them. Subdomains without certificates (HTTP-only services, internal DNS records not tied to certificates) won't appear. For most modern domains, CT logs cover the majority of active subdomains since HTTPS adoption is widespread.

How is this different from DNS brute-forcing?

DNS brute-forcing tries thousands of common subdomain names and checks if they resolve. CT-based discovery is passive — it reads public certificate logs without sending any traffic to the target domain. CT discovery finds actual subdomains that exist (or existed) regardless of naming convention, while brute-forcing only finds subdomains that match its wordlist.

What does 'First Seen' and 'Last Seen' mean?

First Seen is the earliest certificate validity start date (not_before) found for that subdomain. Last Seen is the latest certificate expiry date (not_after). These dates show the time range during which certificates were active for the subdomain.

Can I use this for bug bounty or penetration testing?

Yes. CT-based subdomain enumeration is a standard reconnaissance technique used in authorized security testing. The data is entirely public — anyone can query CT logs. Always ensure you have proper authorization before testing any discovered subdomains.
